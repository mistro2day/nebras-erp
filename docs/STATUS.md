## الأربعة أسئلة — إجابات مباشرة

### ✅ يعمل فعلاً (طرف‑لطرف)
- **المصادقة** — دخول/أدوار/مستأجرين
- **الطلاب** — أعمق وحدة مربوطة
- **المالية/المحاسبة** — واجهة متعددة الصفحات + seed
- **رسوم الطلاب** — توفير تلقائي + رؤية 360°
- **الامتحانات** — مربوطة + بيانات مبذورة
- **المراسلات + واتساب** — Evolution API فعلي

### ⚠️ ناقص / يحتاج عملاً
- توحيد **المعلمين** (3 تطبيقات متفرقة)
- إتمام محرك **التقارير** — إزالة الـmock
- **الفوترة واشتراكات SaaS**
- ربط **admissions / crm / portal** ببياناتها الحقيقية

### ⬜ واجهات فقط — لا تلمس الخلفية
`command` · `config` · `documents` · `errors` · `forms` · `integration` · `personalization` · `placeholder`

### 🔴 بيانات وهمية مثبّتة (hardcoded)
`portal` · `admissions` · `crm` · `command` · `integration` · `finance` · `config`
> تعرض أرقاماً وهمية حتى لو كان الـAPI جاهزاً خلفها.

## الخطوة العملية المقترحة

نختار وحدة واحدة «شبه مكتملة» — **admissions** أو **المعلمون** — ونتتبّعها من الواجهة ← API ← قاعدة البيانات، لنُثبت أين الحلقة المكسورة بالضبط ثم نُصلحها. الإصلاح المتسلسل وحدةً وحدةً يُخرجنا من التيه.

## منهجية القياس

- **عمق الخلفية** = عدد أسطر طبقة الـdomain لكل تطبيق (`backend/apps/*/`).
- **الربط** = عدد ملفات الواجهة التي تستدعي `HttpClient` / `ApiClient` / `apiUrl` / `/api/v1` لكل وحدة (`frontend/src/app/features/*/`).
- **البيانات الوهمية** = مصفوفات مثبّتة داخل ملفات `.ts` (`= signal([...])` / `= [{...}]`).
