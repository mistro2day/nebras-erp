- **الفوترة واشتراكات SaaS**
- ربط **crm / portal** ببياناتها الحقيقية

## وحدة القبول والتسجيل — مكتملة طرف‑لطرف

> **أُنجزت:** 23 يوليو 2026 · أول وحدة تُصلَح بالكامل ضمن خطة الإصلاح المتسلسل.
> المتطلبات الميدانية مستمدة من وثائق مدرسة حقيقية (مستأجر نموذجي — لا اسم النظام).

### سير العمل المعتمد (5 مراحل)

```
مُقدّم ──[قبول الطلب]──> مؤهّل للقدرات ──[رصد الدرجات]──> رُصدت الدرجات
   └──[قرار نهائي]──> مقبول / مرفوض / انتظار ──[تسجيل كطالب]──> مُسجّل
```

**نقطة جوهرية:** «قبول الطلب» ≠ القبول النهائي. الأول يؤهّل المتقدم لدخول امتحان القدرات فقط؛ القبول النهائي يأتي بعد رصد الدرجات وتقييمها.

### الحالات على الخادم

| الحالة | المعنى |
| ------ | ------ |
| `submitted` / `under_review` | مُقدّم (تشمل مراجعة المدرسة) |
| `qualified_exam` | مؤهّل لامتحان القدرات *(جديدة)* |
| `exam_scored` | رُصدت درجات القدرات *(جديدة)* |
| `accepted` / `rejected` / `waitlist` | القرار النهائي |
| `enrolled` | مُسجّل كطالب |

> أُعيد استخدام `accepted` للقبول النهائي للحفاظ على قالب واتساب `ADM_ACCEPTED` ومنطق تحويل الطالب.

### نقاط النهاية المضافة (`apps/admissions`)

| المسار | الوظيفة |
| ------ | ------- |
| `POST applicants/:id/qualify-exam/` | «قبول الطلب» — يؤهّل للقدرات وينشئ صفوف المواد تلقائياً |
| `POST applicants/:id/record-aptitude/` | رصد الدرجات `{scores:[{subject,marks}]}` ويحسب نجاح كل مادة |
| `GET applicants/:id/aptitude-evaluation/` | تقييم محسوب + قرار مقترح (لا يغيّر الحالة) |

### الإعدادات القابلة للتحكم (`AdmissionSettings`)

- **مواد القدرات وعتباتها**: `aptitude_subjects` (اسم/قصوى/عتبة) + `aptitude_total_pass`
  - افتراضياً: اللغة العربية · الإنجليزية · الرياضيات · الكيمياء (عتبة 75 من 100)
- **الرسوم المعروضة**: `registration_fee` · `annual_tuition` · `fee_currency` · `fee_installments` · `fee_notes`
  - **مصدر واحد** يغذّي: استمارة التقديم العامة + الاستمارة الورقية للطباعة + صفحة الإعدادات
  - القيم الافتراضية منقولة من الاستمارة الرسمية إلى ثابت `DEFAULT_ADMISSION_FEES`
  - **لا أرقام مثبّتة في الكود** — تتغيّر لكل فرع وعام من الإعدادات

### الواجهة

- **الأزرار المبعثرة → stepper مراحل** + شريط إجراء واحد للمرحلة الحالية مع تلميح الخطوة التالية
- **لوحة امتحان القدرات**: إدخال درجة لكل مادة (تلوين نجاح/رسوب فوري) + بطاقة التقييم المحسوب
- تحويل معرّفات الصف/العام (UUID) إلى أسماء مقروءة في العرض والطباعة

### إصلاحات جانبية أثناء العمل

| الإصلاح | الأثر |
| ------- | ----- |
| نقل أولياء الأمور عند التحويل لطالب | كانت `StudentFamilyRelation` تبقى فارغة رغم إدخال البيانات في الاستمارة |
| `tenant_id` read-only في سيريلايزر القوالب | كان يُفشل **كل** إنشاء قالب رسائل بخطأ 400 |
| إصلاح `@media print` في الاستمارة | كان المحتوى يُقتطع لصفحة واحدة بسبب قيود المودال |
| فاتورة رسوم التسجيل تلقائياً | `BillingService.bill_new_student_registration` عند التسجيل — **best-effort** لا يُفشل التسجيل عند نقص الإعداد المالي |

## الخطوة العملية التالية

**وحدة المعلمين** — لكنها تبدأ بقرار معماري لا بربط: الكيان مكرّر حرفياً في تطبيقين
(`TeacherProfile` و`TeacherAssignment` موجودان في `teachers` **و** `faculty`)، بالإضافة إلى نموذج `Employee` الغني في `employees`.
الخطوة الأولى: تحديد مصدر الحقيقة وتوحيد النماذج، ثم الربط طرف‑لطرف كما فُعل بالقبول.

## منهجية القياس

- **عمق الخلفية** = عدد أسطر طبقة الـdomain لكل تطبيق (`backend/apps/*/`).
- **الربط** = عدد ملفات الواجهة التي تستدعي `HttpClient` / `ApiClient` / `apiUrl` / `/api/v1` لكل وحدة (`frontend/src/app/features/*/`).
- **البيانات الوهمية** = مصفوفات مثبّتة داخل ملفات `.ts` (`= signal([...])` / `= [{...}]`).
